## What Cetele is

A mobile-first **group dhikr tracker** (installable PWA) that uses dopamine hooks + real group accountability to make daily dhikr stick. Full spec: [`docs/PRD.md`](../docs/PRD.md).

Push notifications = v1.1 fast-follow                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| D8  | Retention = dopamine layer **on top of** group accountability + "never miss twice" forgiveness                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| D9  | **Three roles**: `member` · `group_admin` (per group) · `admin` (app-level, via `users.is_admin`)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| D10 | Notifications = **Web Push + service worker** (VAPID, sent from Vercel cron / Supabase Edge Fn); **email fallback** (Resend) for non-installers / iOS <16.4. Push is v1.1                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| D11 | **Brand = Youth Nexus** (navy `#1D3A5F` + orange `#F26522`), replacing the placeholder green. Design system tokens in `app/globals.css`, components in `components/ui/`, reference at `/designsystem`, guidelines in `docs/DESIGN_SYSTEM.md`                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| D12 | UI primitives are **bespoke** (cva + tailwind-merge) rather than shadcn — same underpinnings, so shadcn/Radix can be layered in later for complex widgets (dialogs, dropdowns)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| D13 | Colours grounded in psychology research: navy=focus/trust surface, orange=arousal for action/celebration only, red=errors only. **`accent-foreground` is navy not white** (white-on-orange fails WCAG AA ~3.1:1; navy ~5.0:1). Soft off-white surface for calm. Never colour-alone (glyph/label + colour). **Amended (CET-15): green promoted to a semantic — `green = completion / streak-alive / growth` (closed rings, "goal met", done-states).** This is both the correct retention signal _and_ the Islamic-green resonance a dhikr audience expects, while navy+orange stay the brand (research-validated as the optimal trust+action pairing). See `docs/UI_PRACTICES.md` + `docs/DESIGN_SYSTEM.md` §Colour psychology |
| D14 | **Token contract, lint-enforced.** All UI values are CSS-variable tokens in `app/globals.css` (colour, type scale, weights, spacing, radii, shadows, motion, z-index/layout). Components reference utilities or `var(--token)` — hardcoded hex/`rgb()`/`hsl()` is an ESLint error. Sole sanctioned literal: `lib/brand.ts` (`BRAND_THEME_COLOR`, for PWA/meta APIs that can't read CSS vars)                                                                                                                                                                                                                                                                                                                                   |
| D15 | **Formatting = Prettier** (+ `prettier-plugin-tailwindcss` class sorting), enforced via lint-staged (pre-commit) and CI `format:check`. Config in `.prettierrc.json` / `.prettierignore`; `eslint-config-prettier` defers all formatting rules to Prettier.                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| D16 | **Branches yes, PRs optional (solo).** Standard flow = a **feature branch per Linear issue** (`mohidkhanzada/cet-N-…`, matching Linear's branch name); merge to `main` yourself once green. **PRs are not institutionalised** — solo means self-approval, so opening a PR is ad-hoc, only when a Vercel preview or review trail is wanted (every branch push gets a preview regardless). **husky pre-commit = the gate** (blocks locally), **GitHub CI = the backstop** (post-push). `gh` installed for when a PR _is_ wanted.                                                                                                                                                                                                 |
| D17 | **Generic vocabulary; brand stays "Cetele".** The trackable item is a **`task`** (replaces `dhikr_items`) — `tasks(id, group_id, label, subtitle, target_count, sort_order)`, where `subtitle` (optional) replaces `arabic`; logs FK is `task_id`. UI **and** code use generic terms (tasks, counts, groups), not "dhikr". **Cetele** remains the product name. Docs/positioning keep **dhikr communities as the primary audience** (dhikr = flagship use case; the software itself is generic). Applies from CET-2 onward.                                                                                                                                                                                                    |
| D18 | **Mock-first validation before backend (CET-14).** Build a fully clickable, frontend-only prototype (zero Supabase/auth/RLS) to put in front of the product owner and gauge reactions **before** spending on backend. All state is faked in-memory + `localStorage` (`lib/mock/*`); a `setInterval` simulates Realtime. Screens cover every role (member / group_admin / app admin) and v1 feature. Requirement changes from the demo feed back into the PRD; **CET-2 (Supabase) is deferred until requirements lock.** Types deliberately mirror PRD §6 so the real schema maps 1:1 behind the same components.                                                                                                               |
| D19 | **Responsive + theming + research (CET-15).** App is **responsive** (mobile = single column + bottom tab bar; ≥`lg` = persistent left sidebar + wider/2-col content), not a phone-only cage. **Light/dark theming** = system default + persisted manual toggle (System/Light/Dark) via a `.dark` class on `<html>` set before paint (no FOUC); dark hues desaturated/lightened per research. Colour psychology, theming, and responsive practices researched + written to **`docs/UI_PRACTICES.md`** (arabic-app style: findings → rules to hold → sources).                                                                                                                                                                   |
